# Telegram Content Access Demo Bot

[English version](README.md)

Python-бот для временного доступа к видеоурокам в Telegram.

Проект показывает рабочую механику для инфопродуктов, онлайн-курсов, закрытых уроков и других Telegram-продуктов, где нужно не просто отправить пользователю видео, а управлять доступом к материалу после выдачи.

Ключевая идея:

```text
не отправлять урок как одноразовое сообщение,
которое потом нужно удалить,

а использовать управляемое сообщение-контейнер,
которое можно переводить между состояниями.
```

## Работающий бот

[Открыть демо-бота в Telegram](https://t.me/info_access_demo_bot)

Бот показывает основной сценарий: видео-заглушка → проверка доступа → временное открытие урока → возврат к заглушке по таймеру → повторное открытие при активном доступе.



## Какую проблему решает бот

В Telegram-ботах для инфобизнеса часто возникает задача:

* открыть пользователю доступ к уроку;
* проверить, есть ли у него право доступа;
* показать материал только при активном доступе;
* закрыть доступ после окончания сессии;
* предложить продление, если доступ закончился;
* не засорять чат новыми копиями одного и того же материала;
* не полагаться на удаление старых сообщений.

Наивная схема выглядит так:

```text
отправить пользователю видеоурок
→ подождать окончания доступа
→ удалить сообщение
```

Но такая модель хрупкая: удаление сообщений в Telegram имеет ограничения и не подходит как единственная опора для управления доступом к контенту.

В этом проекте используется другая модель:

```text
видео-заглушка
→ проверка доступа
→ видеоурок
→ таймер
→ возврат к заглушке
```

Материал остаётся в одном управляемом сообщении, а бот меняет его содержимое в зависимости от состояния доступа.

## Что делает бот

Бот демонстрирует механику временного доступа к одному видеоуроку.

Реализовано:

* главное меню;
* имитация группы доступа;
* кнопка «Войти в группу»;
* кнопка «Выйти из группы»;
* материал с видео-заглушкой;
* кнопка «Посмотреть урок»;
* проверка доступа перед показом урока;
* подмена видео-заглушки на основной видеоурок;
* автоматический возврат к заглушке по таймеру;
* ручное скрытие урока;
* отказ при отсутствии доступа;
* предложение продлить доступ;
* админ-команда для получения `file_id` видео;
* защита админ-команд через `ADMIN_USER_ID`;
* Docker-деплой;
* конфигурация через переменные окружения.

## Основной сценарий

### Пользователь с доступом

```text
/start
→ Войти в группу
→ Открыть материал
→ бот показывает видео-заглушку
→ Посмотреть урок
→ бот проверяет доступ
→ заглушка заменяется на урок
→ запускается таймер
→ после таймера урок возвращается в заглушку
```

### Пользователь без доступа

```text
/start
→ Открыть материал
→ бот показывает видео-заглушку
→ Посмотреть урок
→ бот проверяет доступ
→ доступа нет
→ урок не показывается
→ бот предлагает продлить доступ
```

## Модель состояний

### Состояния доступа пользователя

```text
NO_ACCESS  — доступа нет
HAS_ACCESS — доступ активен
```

### Состояния материала

```text
LOCKED  — материал закрыт, показана заглушка
OPENED  — материал открыт, показан урок
EXPIRED — сессия просмотра завершена, снова показана заглушка
```

## Почему используется видео-заглушка

Видео-заглушка — это безопасное состояние материала.

Пользователь видит, что урок существует, но сам контент не открыт. Когда доступ подтверждён, бот заменяет заглушку на основной видеоурок. Когда сессия заканчивается, бот возвращает материал обратно к заглушке.

Такой подход позволяет:

* не отправлять новый урок каждый раз;
* не удалять старые сообщения;
* управлять доступом через состояние сообщения;
* показывать пользователю понятный интерфейс;
* добавлять продление доступа в тот же сценарий.

## Команды

```text
/start
```

Открывает главное меню.

```text
/whoami
```

Показывает `user_id` и `chat_id` текущего пользователя.

Используется для настройки администратора.

```text
/upload
```

Админ-команда для загрузки видео и получения Telegram `file_id`.

Доступна только пользователю, чей `user_id` указан в переменной окружения `ADMIN_USER_ID`.

## Переменные окружения

Пример конфигурации находится в `.env.example`.

```env
# Telegram bot token from BotFather
BOT_TOKEN=

# Telegram user_id of the admin
ADMIN_USER_ID=

# Telegram file_id of the placeholder video
PLACEHOLDER_VIDEO_ID=

# Telegram file_id of the lesson video
LESSON_VIDEO_ID=

# Lesson access duration in seconds
# Max: 172740 seconds = 47 hours 59 minutes
LESSON_TTL_SECONDS=30
```

### BOT_TOKEN

Токен Telegram-бота, полученный через BotFather.

Не должен храниться в репозитории.

### ADMIN_USER_ID

Telegram `user_id` администратора.

Администратор может отправлять видео боту и получать их `file_id`.

### PLACEHOLDER_VIDEO_ID

Telegram `file_id` видео-заглушки.

Это видео показывается, когда урок закрыт, недоступен или сессия просмотра завершена.

### LESSON_VIDEO_ID

Telegram `file_id` основного видеоурока.

Это видео показывается только пользователю с активным доступом.

### LESSON_TTL_SECONDS

Длительность открытой сессии просмотра в секундах.

После истечения этого времени бот автоматически возвращает сообщение к видео-заглушке.

Максимальное значение:

```env
LESSON_TTL_SECONDS=172740
```

Это 47 часов 59 минут.

## Где хранятся видео

Видео не хранятся в репозитории и не загружаются в Docker-контейнер.

Схема работы такая:

```text
администратор отправляет видео боту
→ Telegram сохраняет медиа
→ бот возвращает file_id
→ file_id добавляется в переменные окружения
→ бот использует file_id для отправки и редактирования видео
```

Преимущества:

* видео не попадают в GitHub;
* контейнер остаётся лёгким;
* не нужен отдельный storage;
* не нужно хранить медиафайлы на сервере;
* бот работает с Telegram-hosted media.

Важно: `file_id` нужно получать для конкретного бота. Если создать нового Telegram-бота, видео нужно отправить ему заново и получить новые `file_id`.

## Как настроить видео

1. Запустить бота.
2. Выполнить команду:

```text
/whoami
```

3. Скопировать `user_id`.
4. Добавить его в переменную окружения:

```env
ADMIN_USER_ID=your_telegram_user_id
```

5. Перезапустить контейнер.
6. Выполнить команду:

```text
/upload
```

7. Отправить боту видео-заглушку.
8. Скопировать полученный `file_id` в переменную:

```env
PLACEHOLDER_VIDEO_ID=
```

9. Отправить боту основной видеоурок.
10. Скопировать полученный `file_id` в переменную:

```env
LESSON_VIDEO_ID=
```

11. Перезапустить контейнер.

После этого бот готов к проверке основного сценария.

Задать переменные окружения:

Запустить бота:

```bash
python main.py
```

## Запуск через Docker

Собрать образ:

```bash
docker build -t telegram-content-access-demo .
```

Запустить контейнер:

```bash
docker run --rm \
  -e BOT_TOKEN="your_bot_token" \
  -e ADMIN_USER_ID="your_telegram_user_id" \
  -e PLACEHOLDER_VIDEO_ID="placeholder_file_id" \
  -e LESSON_VIDEO_ID="lesson_file_id" \
  -e LESSON_TTL_SECONDS=30 \
  telegram-content-access-demo
```

## Ограничения текущей версии

Текущая реализация показывает механику временного доступа к одному видеоуроку.

В демо-версии:

* один материал;
* нет базы данных;
* доступы хранятся в памяти;
* состояние сбрасывается при перезапуске контейнера;
* таймеры хранятся в памяти;
* кнопка «Продлить доступ» показывает сценарий, но не подключена к оплате.

Эти ограничения не мешают проверить основную механику: бот открывает урок только при активном доступе и автоматически возвращает материал к заглушке после окончания сессии просмотра.

## Практическая польза

Подход решает одну из типовых проблем Telegram-продуктов: как временно открыть доступ к материалу и затем закрыть его без удаления сообщения.

Вместо схемы:

```text
отправить видео → потом удалить сообщение
```

используется схема:

```text
заглушка → проверка доступа → урок → таймер → заглушка
```

Это даёт несколько практических преимуществ:

* пользователь не получает постоянную копию урока в чате;
* чат не засоряется повторными отправками одного и того же материала;
* материал остаётся в одном управляемом сообщении;
* доступ можно проверять перед каждым открытием;
* после окончания сессии пользователь видит понятное закрытое состояние;
* в тот же сценарий можно встроить продление доступа;
* механику можно использовать для платных уроков, записей эфиров, бонусов и временных материалов.

## Для каких задач подходит

Механика подходит для Telegram-ботов, которые обслуживают:

* онлайн-курсы;
* закрытые уроки;
* платные записи эфиров;
* временный доступ к материалам;
* продления доступа;
* тестовые фрагменты уроков;
* бонусные материалы после оплаты;
* закрытые клубы и группы;
* инфопродукты с ограниченным сроком просмотра.

## Что демонстрирует проект

Проект показывает рабочий паттерн управления доступом к Telegram-контенту:

* материал не удаляется, а переводится между состояниями;
* доступ проверяется перед открытием урока;
* видеоурок временно подменяет заглушку;
* после таймера сообщение возвращается к закрытому состоянию;
* медиафайлы не хранятся в репозитории или контейнере;
* настройки вынесены в переменные окружения;
* бот разворачивается как Docker-контейнер.
